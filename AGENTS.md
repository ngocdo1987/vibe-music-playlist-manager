# Repository Guidelines

## Project Structure & Module Organization

This repository is a pure Zig web app for managing MP3 playlists. Source lives under `src/`: `main.zig` starts the TCP server, `router.zig` parses HTTP and dispatches routes, `db.zig` owns SQLite access, and `auth.zig` loads credentials. HTML is generated by Zig functions in `src/templates/`; reusable helpers live in `src/utils/`. Uploaded audio is stored in `public/mp3/` and the SQLite database is created at runtime. Build metadata is in `build.zig` and `build.zig.zon`; container support is in `Dockerfile` and `docker-compose.yml`.

## Build, Test, and Development Commands

- `zig build` builds `zig-out/bin/music-manager`.
- `zig build run` builds and runs the local server, normally at `http://localhost:8080`.
- `zig build test` runs the configured unit test step from `src/main.zig`.
- `zig test src/router.zig` or `zig test src/utils/mp3.zig` runs focused module tests.
- `zig fmt src/**/*.zig build.zig` formats Zig sources before review.
- `docker-compose up -d` builds and starts the containerized app.

The app links against system SQLite, so install `sqlite3` and `libsqlite3-dev` on Linux before building locally.

## Coding Style & Naming Conventions

Follow standard Zig formatting with `zig fmt`; use 4-space indentation and keep names descriptive. Files and modules use lowercase names such as `multipart.zig`; public types use PascalCase; functions and variables use lower camel case. Prefer explicit error handling and bounded buffers consistent with the current router/template code. Keep generated HTML in `src/templates/` and avoid introducing a template engine unless the architecture changes intentionally.

## Testing Guidelines

Add tests near the module they exercise using Zig `test` blocks. Prefer focused tests for parsing, validation, database behavior, and request routing. Name tests by behavior, for example `test "rejects non-mp3 upload"`. Run `zig build test` before opening a PR; run specific `zig test ...` commands while iterating.

## Commit & Pull Request Guidelines

Recent history uses short imperative commit subjects, for example `Add AI chat` and `Fix wrong Markdown format`. Keep commits focused and avoid mixing unrelated refactors with feature work. Pull requests should include a brief summary, test commands run, any configuration changes, and screenshots or notes for visible UI changes.

## Security & Configuration Tips

Configure `ADMIN_USERNAME`, `ADMIN_PASSWORD`, and `SERVER_PORT` in `.env` or Docker environment variables. Do not commit secrets, uploaded MP3 files, `music_manager.db`, or generated build output. Validate upload paths and file types carefully; `public/mp3/` is user-facing storage.
