# Repository Guidelines

## Project Structure & Module Organization

This is a Rust Actix Web application for managing MP3 playlists with SQLite persistence and Tera-rendered HTML. Core application code lives in `src/`: `main.rs` wires configuration and routes, `db.rs` handles database access, `models.rs` defines shared data structures, and `src/handlers/` contains route handlers split by area (`admin.rs`, `auth.rs`, `api.rs`, `public.rs`). HTML templates live under `templates/`, grouped by `admin/`, `auth/`, and `public/`. Uploaded MP3 files are expected in an `mp3/` runtime directory, which should not be committed.

## Build, Test, and Development Commands

- `cargo run`: start the development server using local environment variables.
- `cargo build`: compile the application in debug mode.
- `cargo build --release`: produce an optimized binary at `target/release/music-manager`.
- `cargo test`: run Rust unit and integration tests when present.
- `cargo fmt`: format Rust source using rustfmt.
- `cargo clippy`: run additional Rust lint checks before larger changes.

Configure local runs with `.env` values such as `ADMIN_USERNAME`, `ADMIN_PASSWORD`, `HOST`, `PORT`, and `SESSION_SECRET`.

## Coding Style & Naming Conventions

Use Rust 2021 idioms and keep code formatted with `cargo fmt`. Follow standard Rust naming: `snake_case` for functions, modules, variables, and files; `PascalCase` for structs and enums; `SCREAMING_SNAKE_CASE` for constants. Keep handlers grouped by feature area, and prefer small helper functions when route logic grows. Tera templates should use descriptive block and file names that match the route or page they render.

## Testing Guidelines

No dedicated test suite is currently present. Add tests alongside new behavior, using Rust unit tests in the relevant source file or integration tests under `tests/` for HTTP and database flows. Name tests after the behavior, for example `creates_playlist_with_ordered_songs`. Run `cargo test` before opening a pull request, and include manual verification notes for template or upload changes.

## Commit & Pull Request Guidelines

Recent commits use short imperative messages such as `Fix wrong Markdown format` and `Add AdonisJS docs`. Keep commit subjects concise, action-oriented, and specific. Pull requests should explain the change, note any database or environment impacts, list test results, and include screenshots for admin or public UI changes.

## Security & Configuration Tips

Do not commit `.env`, SQLite database files, uploaded MP3s, or generated `target/` artifacts. Use a strong `SESSION_SECRET` such as one generated by `openssl rand -hex 32`. Validate uploads carefully and keep public file-serving paths constrained to expected media directories.
